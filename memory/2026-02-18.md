### Button Flow (IMPLEMENTED & TESTED)
1. calendar-agent calls `send-buttons.js` to get formatted message with buttons
2. Message sent to @OpenclawSam with inline_keyboard
3. User clicks button (e.g., "1 hour")
4. Telegram sends callback_data (cs|taskId|duration|1hr)
5. Plugin intercepts, updates SQLite, edits message with next buttons
6. Agent checks SQLite, continues workflow
7. Repeat for type (online/in-person), then confirm
8. On confirm, calendar-agent creates real event in Google Calendar

### Architecture Diagram (Final)
```
@OpenclawSam (Telegram bot)
  ↓
Binding: channel=telegram → master-agent
  ↓
master-agent recognizes "schedule"
  ↓
Routes to calendar-agent
  ↓
calendar-agent:
  1. Find contact (contact-tool.js search)
  2. Check calendar (calendar-tool.js checkConflicts)
  3. Call send-buttons.js → get JSON with buttons
  4. Send message to Telegram with inline_keyboard
  5. User clicks button
  6. Plugin intercepts (telegram-button-plugin.js)
  7. Plugin updates SQLite
  8. Plugin edits message with next buttons
  9. Repeat steps 5-8 for duration, type, confirm
  10. calendar-agent reads SQLite
  11. Creates event (calendar-tool.js createEvent)
  12. Returns confirmation
```

### Lesson Learned
- One OpenClaw instance with routing is simpler than two instances
- Button interception needs plugin + database coordination
- LLM agents can't directly send Telegram messages, need helper tools
- System works best when agent calls CLI tools (send-buttons.js, etc)
